# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Software Factory is an automated development pipeline where AI agents handle the full cycle from requirements to merge. A user qualifies a need via a chat interface → Claude creates GitLab epics/issues → a Temporal workflow orchestrates a sequence of agents: dev agent (code generation), review agent, review-fix agent, static analysis agent (SonarQube), and final merge.

**Reference stack:**

| Component | Tool | Mode |
|---|---|---|
| Ticketing / VCS / CI-CD | GitLab CE | On-prem (Docker) |
| Static analysis | SonarQube Community Edition | On-prem (Docker) |
| Workflow orchestration | Temporal | On-prem (Docker) |
| LLM | Claude (Anthropic API) | Cloud |
| Agent ↔ tool communication | MCP servers | On-prem |

**Application language: TypeScript (strict mode)**

All application code (MCP servers, agents, Temporal workers, backend) must be written in TypeScript — never vanilla JavaScript. Use `tsc --strict` for all packages. Rationale: `@modelcontextprotocol/sdk` and the Temporal TypeScript SDK are the most complete SDKs for their respective tools, and strict TypeScript provides compile-time guarantees that Python type hints and vanilla JS cannot match.

**Frontend framework: Vue.js**

All frontend/UI work must use Vue.js (Vue 3 Composition API) as the base framework. Do not use React or Next.js unless explicitly requested.

**MCP servers: HTTP transport, containerized**

Every MCP server must be deployed as its own Docker container in `infrastructure/docker-compose.yml` and joined to `factory-network`. All MCP communication uses the HTTP transport (Streamable HTTP) — never stdio. Agents connect to MCP servers via HTTP URL (e.g. `http://mcp-gitlab:3000/mcp`). Do not use `StdioServerTransport` anywhere in the codebase.

## General tooling rules

**GitLab project:** When using the GitLab MCP server from this repository, the target project is **"Software Factory"**, ID `3`.

**Do not use the `gh` CLI.** To fetch information about GitHub repositories (README, releases, file content, API data), use HTTP directly — e.g. `curl https://api.github.com/...` or `fetch()` in TypeScript. The `gh` command is not guaranteed to be installed or authenticated in this environment.

## Infrastructure

All services run via Docker Compose. The shared Docker network `factory-network` must exist before starting services individually.

```bash
# Start all current services (GitLab CE + runner + SonarQube + PostgreSQL)
docker compose -f infrastructure/docker-compose.yml up -d

# Stop
docker compose -f infrastructure/docker-compose.yml down

# Bootstrap GitLab (run once after first `up`): creates test project, registers runner, validates CI
bash infrastructure/scripts/setup-gitlab.sh

# Bootstrap SonarQube (run once after first `up`): checks vm.max_map_count, secures admin account
bash infrastructure/scripts/setup-sonarqube.sh

# Bootstrap Temporal (run once after first `up`): creates factory-test namespace, verifies worker
bash infrastructure/scripts/setup-temporal.sh

# Bootstrap inter-service network (run after all other setup scripts): validates connectivity,
# sets CI/CD variables, updates .gitlab-ci.yml with SonarQube + Temporal stages
bash infrastructure/scripts/setup-network.sh
```

**Host prerequisite for SonarQube** (Elasticsearch requires this — set once per machine):
```bash
sudo sysctl -w vm.max_map_count=524288
# Persist across reboots:
echo 'vm.max_map_count=524288' | sudo tee /etc/sysctl.d/sonarqube.conf
```

Copy `infrastructure/.env.example` to `infrastructure/.env` and fill in values before starting. The `.env` file is gitignored.

`setup-gitlab.sh` is idempotent — safe to re-run. It waits up to 10 minutes for GitLab to become healthy before proceeding.

## Epics and user stories

- `docs/epic.md` — all 10 epics with scope, dependencies, and done criteria
- `docs/user_stories_N.md` — user stories for epic N (generated by `/gen-us`)
- `.claude/backlog.md` — moderate-severity items deferred during implementation (format: `- [US-N] description`)

**Notation:** `us: N-X` désigne la user story X de l'épopée N (ex: `us: 1-2` → epic 1, US-2).

Epic dependency chain: EPIC-01 (infra) → EPIC-02 (MCP layer) → EPIC-03 (requirements UI) / EPIC-04 (Temporal orchestration) → EPIC-05–09 (agents) → EPIC-10 (observability).

## Slash commands

| Command | Purpose |
|---|---|
| `/gen-us <N>` | Generate user stories for epic N into `docs/user_stories_N.md` |
| `/implement-us <N> <US-X>` | Implement one user story from epic N |
| `/implement-epic <N>` | Implement all user stories in epic N sequentially |
| `/ticket` | Qualify and document a bug or behavior change into `docs/tickets/TICKET-NNN-<slug>.md` |
| `/implement-ticket <NNN>` | Implement ticket NNN (moves file through `docs/tickets/` → `in_progress/` → `done/`) |

## Implementation workflow

All implementation commands (US and tickets) follow the same 5-step cycle defined in `.claude/workflows/us-implementation.md`:

1. **Plan** — enter plan mode, analyze acceptance criteria, produce an ordered action list
2. **Critique** — delegate plan review to an independent `Plan` subagent; classify findings as Grave / Modéré / Esthétique
3. **Revise** — fix Grave issues in the plan; add Modéré items to `.claude/backlog.md`
4. **Implement** — execute the revised plan
5. **Verify** — run lint, check each acceptance criterion explicitly as satisfied/not satisfied

> **Note:** `.claude/workflows/us-implementation.md` contains stack-specific context (Electron + Vue 3 + SQLite) inherited from a prior project. Update that section when implementing epics that introduce a real application stack.
